Iris / analysis report example

Report Before Decoration

Iris should turn thinking into a reading path before it turns anything into a page.

AI output often arrives as a useful but shapeless transcript. Iris should extract the argument, name the evidence path, and make the remaining uncertainty visible enough for a human reader to trust the page.

ThesisA strong Iris report is an edited judgment with evidence, not a decorated dump of model output.

Key findings

01The report starts with a judgment, not a transcript.
02Draft notes are private working material, not the final surface.
03Figures appear only when they clarify a relationship.
01 / Material

The source is material, not the final structure.

Useful inputs can be messy: codebase notes, product critique, research excerpts, event timelines, rough thinking, or a requirements draft.

The agent's first job is to decide what the material is trying to help the reader understand. A late conclusion can become the opening thesis. A scattered set of caveats can become the assumptions section. A long list can become one diagram if the relationship matters more than the wording.

The report should preserve meaning without preserving clutter. It should separate observation from inference and make uncertainty explicit instead of smoothing it away.

Evidence note

Labels, source notes, and citations should use the report language. External references such as HTML anchor guidance open outside the report.

Margin note

Margin notes qualify the argument. They should feel like part of the page's evidence system, not a detached widget.

02 / Voice

The writing should sound like a person thinking clearly.

Iris should avoid filler phrases, false neutrality, and mechanical summary language.

The page can be polished without becoming evasive. It should say what the material supports, what it does not support, and what decision the reader can reasonably make next.

The tone should adapt to the user's language. Chinese reports should read as natural Chinese, not translated English. English reports should avoid swollen product language. Technical reports should stay precise without turning every sentence into jargon.

Assumptions

  • An agent has already read the source material.
  • The report is for human decision support.

Anti-patterns

  • Do not fill space with generic summary language.
  • Do not invent certainty when the source only supports a question.
